# Goal Resolution Model

For the contributor-facing planning contract, read `docs/PLANNING_SOURCE_OF_TRUTH_OVERVIEW.md` alongside this model.

## Purpose

This model defines how the product should turn vague user goal language into structured planning inputs without making AI the system of record.

It sits between intake and canonical planning.

- intake captures what the user said
- AI may interpret, label, and propose missing structure
- the app collects confirmation or correction
- only the user-confirmed resolved goal object becomes canonical planning input

This preserves the core rule from the product architecture: deterministic planning runs on typed application state, not on raw AI output.

## Design Goals

- support vague, mixed, and appearance-driven goals without forcing users into a tiny preset list
- preserve the user's original words
- keep the final planner deterministic
- let AI help with interpretation, not with truth assignment
- stay practical enough to drive `Goal`, `goalState`, `ProgramBlock`, and `PlanWeek`
- avoid creating a second parallel goal system

## Where This Fits

Goal resolution is not itself the planner.

It is a preprocessing step that produces a small, stable, confirmed goal object that the planner can consume. The planning stack remains:

1. user profile
2. resolved goal stack
3. program block
4. weekly intent
5. plan week
6. plan day

The output of this model should feed the canonical goal layer, not bypass it.

## Capability Packet Handoff

Resolved goals now hand off into a `GoalCapabilityPacket` before weekly planning.

That packet is the planner-facing bridge between infinite user phrasing and finite planning logic. It declares:

- dominant domain
- support domains
- capability mix
- confidence
- missing anchors
- fallback planning mode

This keeps goal resolution flexible while keeping the downstream planner deterministic.

## Core Principle

There are three distinct layers:

### 1. Raw intent

What the user actually said.

Example:

```js
{
  rawGoalIntent: "I want to look athletic again and maybe run a 10k this fall."
}
```

### 2. Interpreted hypothesis

A structured proposal generated by AI or deterministic heuristics.

Example:

```js
{
  goalHypothesis: {
    summary: "Mixed body-composition and endurance goal",
    goalFamily: "hybrid",
    measurableTier: "proxy_measurable",
    primaryMetric: null,
    proxyMetrics: ["waist", "bodyweight trend", "10k completion", "easy run consistency"],
    targetHorizon: "fall",
    confidence: "medium"
  }
}
```

This layer is useful, but it is not canonical truth.

### 3. User-confirmed resolved goal

The final typed object accepted or edited by the user. This is the only layer planning should treat as canonical.

## Canonical Resolution Object

The product should treat a resolved goal as a compact object with explicit separation between user input, AI interpretation, and confirmed planning truth.

```js
{
  id: "goal_resolution_1",
  status: "resolved",
  rawIntent: {
    text: "I want to look athletic again and maybe run a 10k this fall.",
    capturedAt: "2026-04-09T18:00:00.000Z"
  },
  hypothesis: {
    source: "ai_interpretation",
    summary: "Mixed body-composition and endurance goal",
    goalFamily: "hybrid",
    measurableTier: "proxy_measurable",
    primaryMetric: null,
    proxyMetrics: [
      { key: "waist", label: "Waist circumference", unit: "in" },
      { key: "bodyweight_trend", label: "Bodyweight trend", unit: "lb" },
      { key: "run_10k_completion", label: "10k completion", unit: "boolean" }
    ],

      "Aggressive fat loss may limit run-quality progression",
      "Race prep may reduce hypertrophy volume"

  },
  resolved: {
    confirmedByUser: true,
    summary: "Improve athletic appearance while building toward a fall 10k",
    goalFamily: "hybrid",
    measurableTier: "proxy_measurable",

      { key: "waist", label: "Waist circumference", unit: "in" },

    refinementTrigger: "new_checkin_or_metric_gap"
  }

```

## Field Definitions

### Raw goal intent in the user's words

- required
- stores the user's original language with no interpretation baked in
- should always remain visible for audit and future refinement

### AI-interpreted goal hypothesis

- optional but expected in AI-assisted intake
- may classify the goal, infer plausible metrics, surface ambiguities, and propose tradeoffs
- must be marked as a hypothesis or proposal
- must never be consumed by the planner unless confirmed into the resolved object

### Measurability tier

Use one of:

- `fully_measurable`
- `proxy_measurable`
- `exploratory_fuzzy`

Interpretation:

- `fully_measurable`: direct target exists and is plan-relevant
  - examples: "run sub-45 10k", "deadlift 405", "lose 15 lb by August 1"
- `proxy_measurable`: direct success is vague but reasonable proxies exist
  - examples: "look athletic", "lean out", "be more fit again"
- `exploratory_fuzzy`: intent is real but too unclear for strong metric-driven planning
  - examples: "get back in shape", "feel like myself again", "become a hybrid athlete"

The tier affects planning strictness:

- fully measurable goals can anchor horizon, progression, and adaptation rules
- proxy measurable goals can influence planning through stable proxy metrics
- exploratory goals should bias toward consistency, capacity rebuilding, and periodic refinement rather than aggressive optimization

### Primary metric

- the single strongest direct measure of success if one exists
- should be absent when no defensible direct metric exists

Examples:

- race time
- event completion
- bodyweight target
- waist measurement
- estimated 1RM
- training frequency adherence

### Proxy metrics

- secondary measurable signals used when direct success is weak or delayed
- should be short, durable, and planner-usable
- avoid large custom metric systems

Good proxy examples:

- weekly run frequency
- long-run duration
- scale trend
- waist trend
- progress photos completed
- pull-up count
- average daily protein adherence

### Target date or horizon

Support either:

- `targetDate`: exact date when known
- `targetHorizonWeeks`: approximate planning horizon when exact date is unknown

The app should not force a fake exact date when only a loose horizon exists.

### Confidence level

Confidence reflects how strong the current resolution is for planning, not how confident the AI sounds.

Use:

- `low`
- `medium`
- `high`

Suggested meaning:

- `low`: goal can guide only broad consistency planning
- `medium`: goal can guide weekly emphasis and tradeoff handling
- `high`: goal is specific enough for direct metric-led planning

### Missing information needed for stronger planning

- captures the smallest set of unanswered questions blocking better planning
- should be action-oriented, not a long questionnaire
- typically limited to 1-3 items

Examples:

- which goal is primary
- exact event date
- realistic current baseline
- acceptable rate of weight loss
- whether appearance or performance takes priority

### Tradeoffs or conflicts with other goals

- records the known tension between this goal and other active goals
- should be explicit because hybrid planning depends on managed compromise

Examples:

- aggressive fat loss vs strength gain
- race-specific volume vs hypertrophy recovery
- physique focus vs performance fueling
- return-to-shape re-entry goals vs ambitious event prep

### User-confirmed resolved goal object

- the only portion that should become canonical planner input
- can be created from direct acceptance, acceptance with edits, or manual user entry
- should be stamped with user confirmation provenance

### Review cadence / refinement trigger

The resolved goal should declare when the system should revisit its clarity.

Review cadence examples:

- `weekly`
- `biweekly`
- `block_start`
- `block_end`
- `monthly`

Refinement trigger examples:

- `goal_confidence_low`
- `missing_metric_data`
- `new_time_bound_event_added`
- `checkin_goal_change_signal`
- `planner_tradeoff_conflict`
- `stalled_proxy_metric`

## Supported Goal Types

The same resolution model should work across all goal styles.

### Performance goals

Example:

- "Run a sub-1:45 half marathon in October"

Typical result:

- measurable tier: `fully_measurable`
- primary metric: race time
- target date: exact

### Strength goals

Example:

- "Get my bench back to 225"

Typical result:

- measurable tier: `fully_measurable`
- primary metric: bench press top set or estimated 1RM
- horizon: estimated if no exact date

### Body composition goals

Example:

- "Lose 12 pounds by summer"

Typical result:

- measurable tier: `fully_measurable`
- primary metric: bodyweight or body-fat proxy
- proxy metrics: waist, compliance, scale trend

### Appearance-based goals

Example:

- "Get abs by summer"

Typical result:

- measurable tier: `proxy_measurable`
- primary metric: often none
- proxy metrics: waist, scale trend, photo check-ins, consistency

### Hybrid or mixed goals

Example:

- "Become a hybrid athlete"

Typical result:

- measurable tier: `exploratory_fuzzy` or `proxy_measurable`
- primary metric: usually none at first
- proxy metrics: strength benchmark, endurance benchmark, weekly training balance
- missing info: what type of hybrid outcome actually matters

### Re-entry goals

Example:

- "Get back in shape"

Typical result:

- measurable tier: `exploratory_fuzzy`
- planning should initially optimize for consistency, capacity rebuilding, and adherence
- review cadence should be tighter so the system can refine once the user shows patterns

## Deterministic Planning Rules

To keep planning deterministic:

1. The planner may only read the user-confirmed `resolved` object.
2. The planner must ignore unconfirmed hypothesis fields.
3. If confidence is low or measurability is fuzzy, planning should shift to conservative defaults rather than invent precision.
4. The planner should use stable enums and typed metric keys, not freeform AI summaries.
5. AI text may help produce the resolution proposal, but deterministic services should map confirmed values into canonical goal records.

In practice:

- `rawIntent` supports audit and future interpretation
- `hypothesis` supports proposal and clarification UX
- `resolved` supports canonical goal creation

## Canonical Mapping To Planning

The resolved goal object should map into the existing canonical goal layer instead of replacing it.

Suggested mapping:

- `resolved.summary` -> goal display name
- `resolved.goalFamily` -> normalized goal category or category mix
- `resolved.targetDate` -> canonical time-bound deadline when present
- `resolved.primaryMetric` / `proxyMetrics` -> tracking metadata
- `resolved.confidence` -> goal-state confidence or resolution confidence field
- `resolved.tradeoffs` -> planning-facing arbitration context
- `resolved.reviewCadence` and `resolved.refinementTrigger` -> intake/planning maintenance metadata

The planner does not need the full resolution object on every pass. It mostly needs the confirmed planning subset.

## Minimal Schema Recommendation

Use a compact schema for the confirmed planning-facing subset.

```js
{
  summary: string,
  goalFamily: "performance" | "strength" | "body_comp" | "appearance" | "hybrid" | "general_fitness",
  measurableTier: "fully_measurable" | "proxy_measurable" | "exploratory_fuzzy",
  primaryMetric: { key: string, unit: string } | null,
  proxyMetrics: [{ key: string, unit: string }],
  targetDate: string,
  targetHorizonWeeks: number | null,
  confidence: "low" | "medium" | "high",
  tradeoffs: string[],
  reviewCadence: string,
  refinementTrigger: string
}
```

This is enough to support planning without turning goal resolution into a sprawling ontology.

## Practical Product Rules

- keep one resolved object per meaningful goal, not one per sentence fragment
- allow mixed-goal resolution when the user clearly expresses combined intent
- ask for clarification only when missing information changes planning materially
- do not force appearance goals into fake precision
- do not force fuzzy goals to stay fuzzy forever; schedule refinement
- do not let AI-generated wording overwrite the user's own intent text
- do not store long AI analysis blobs as canonical data

## Plain-English Flow

1. The user describes what they want in their own words.
2. The system stores that raw wording exactly as given.
3. AI or deterministic logic proposes what the goal probably means, how measurable it is, what metrics might track it, and what is still unclear.
4. The user accepts, edits, or rejects that proposal.
5. The accepted version becomes the resolved goal object.
6. Deterministic planning reads only that confirmed object.
7. If the goal is vague or confidence is low, the planner stays conservative and schedules a review instead of pretending the goal is fully specified.

## Minimum Canonical Fields Needed For Planning

- `summary`
- `goalFamily`
- `measurableTier`
- `primaryMetric`
- `proxyMetrics`
- `targetDate` or `targetHorizonWeeks`
- `confidence`
- `tradeoffs`
- `reviewCadence`
- `refinementTrigger`

## Fields That Should Remain Optional

- `primaryMetric`
- `targetDate`
- `targetHorizonWeeks`
- `missingInformation`
- `tradeoffs`
- `proxyMetrics`
- `hypothesis`

`rawIntent.text`, `resolved.summary`, `resolved.goalFamily`, `resolved.measurableTier`, `resolved.confidence`, and the review/refinement metadata should generally be present even for fuzzy goals.

## 2026-04-13 Hardening Notes

- Natural duration language like `30 minute marathon` must stay attached to the event goal and normalize into a faithful time metric.
- Strength parsing now prefers the intended benchmark when multiple weights are present in the same sentence, which protects phrases like `bench 225, not 45 lb dumbbells`.
- Raw goal intent remains inspectable alongside normalized interpretation so semantic drift can be audited instead of hidden.
